# Mobile Card Title Improvement Plan

## 问题

当前 `mobile_capture_asset_card_v1` 的正文固定以：

```md
# 这是什么
```

开头。

搜索、详情和星球探索系统又会高度依赖 `title` 做人类可读呈现。

因此当前标题推导退化成：

- `这是什么`

这不是有效标题。

## 目标

为手机视觉资产卡提供真正的人类可读标题，并满足：

- 搜索结果可读
- 星球探索展示可读
- 详情页可读
- 不破坏现有正文 block 结构
- 对旧卡保留兼容

## 方案

### 1. 引入 `display_title`

给 `mobile_capture_asset_card_v1` 增加一个显式标题字段：

- `display_title`

写入位置：

- frontmatter
- LanceDB metadata

响应位置：

- 写入接口返回的 `card.title`
- 搜索结果里的 `title`
- 详情接口里的 `title`

### 2. 标题生成规则

写卡时单独生成一个短标题。

要求：

- 尽量概括对象本体
- 尽量带一个关键限定信息
- 简短
- 不能只是“这是什么”
- 不能是空值

推荐风格：

- `Laboratorio Echevarne 检验报告领取说明`
- `1909 年法文宴会菜单`
- `某公司联系人名片`
- `电影海报局部模糊裁切图`

### 3. 正文结构不改

正文仍保留：

- `# 这是什么`
- `# 直接可见信息`
- `# 关键信息提炼`
- `# 限制与风险`

也就是说：

- `display_title` 是卡片标题
- `这是什么` 是正文区块标题

这两个语义分开。

### 4. 搜索侧取值优先级

对于 `mobile_capture_asset_card_v1`：

1. 优先 `display_title`
2. 若没有，则从“这是什么”区块正文第一句提炼短标题
3. 再不行才退回 `Untitled`

对于其他卡类型：

- 保持原有逻辑不变

## 实施顺序

1. 写入服务生成并保存 `display_title`
2. 搜索和详情链路优先读取 `display_title`
3. 增加旧卡兼容回退
4. 补测试并回归

## 不做的事

这轮不做：

- 改正文区块结构
- 让前端自己猜标题
- 对所有 schema 一次性引入统一复杂标题系统
